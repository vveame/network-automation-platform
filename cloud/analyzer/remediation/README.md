# Safe Remediation Layer

This folder contains the controlled remediation layer for the PFE network automation platform.

The remediation layer consumes the final anomaly decision generated by the hybrid rule-based and ML analyzer, then selects a predefined safe action from an allowlist.

It does **not** allow the ML model to directly execute arbitrary commands.

## Purpose

The goal of this module is to close the loop between anomaly detection and controlled remediation.

The platform now follows this flow:

```text
Prometheus metrics
    ↓
Rule-based analyzer
    ↓
ML Isolation Forest analyzer
    ↓
Hybrid final decision
    ↓
Safe remediation runner
    ↓
Allowlisted action only
```

The remediation layer reads:

```text
cloud/analyzer/outputs/final-decision.json
```

and produces:

```text
cloud/analyzer/outputs/remediation/remediation-plan.json
cloud/analyzer/outputs/remediation/remediation-report.txt
```

## Design Principle

The remediation layer is intentionally conservative.

```text
ML-only anomaly = advisory
Rule-based anomaly = explainable anomaly
Rule + ML anomaly = confirmed anomaly
Infrastructure-changing action = requires explicit confirmation
```

The ML model can help detect suspicious behavior, but it cannot directly modify infrastructure.

Automated remediation is allowed only when the final decision contains:

```text
remediation_allowed = true
```

and the user or Jenkins pipeline runs the remediation runner in apply mode with:

```text
--confirm-apply
```

## Safety Policy

The safety policy is defined in:

```text
cloud/analyzer/remediation/safe_actions.json
```

Current policy:

```text
Default mode: plan
Apply mode requires confirmation
ML-only anomalies are advisory
Automated remediation requires rule-based confirmation
```

This prevents uncontrolled remediation and keeps the platform explainable during the PFE demonstration.

## Files

### `safe_actions.json`

Defines the allowlisted remediation actions.

Each action includes:

```text
action name
action type
whether it modifies infrastructure
description
```

The runner only accepts actions listed in this file.

### `run_safe_remediation.py`

Main remediation runner.

It reads the final decision, selects a safe action, and either:

```text
plans the remediation
```

or:

```text
executes it only when apply mode is confirmed
```

Supported modes:

```text
plan
apply
```

Plan mode is the default and does not execute commands.

Apply mode requires:

```text
--confirm-apply
```

## Inputs

The remediation runner consumes:

```text
cloud/analyzer/outputs/final-decision.json
```

This file is produced by:

```text
cloud/analyzer/ml/merge_ml_decision.py
```

The final decision includes:

```text
classification
final_status
final_severity
final_risk_score
recommended_action
remediation_allowed
remediation_mode
rule_anomalous
ml_anomalous
ml_suspicious
decision_reason
```

## Outputs

The remediation runner generates:

```text
cloud/analyzer/outputs/remediation/remediation-plan.json
cloud/analyzer/outputs/remediation/remediation-report.txt
```

### `remediation-plan.json`

Machine-readable remediation result.

It contains:

```text
selected action
execution mode
decision context
whether the action modifies infrastructure
whether the action was executed
success/failure state
exit codes
safety policy
```

### `remediation-report.txt`

Human-readable remediation report.

It contains:

```text
selected action
decision context
rule detection reasons
action description
commands executed or planned
command outputs
```

This file is useful for Jenkins archiving and report screenshots.

## Current Allowlisted Actions

### `no_action`

Type:

```text
noop
```

Infrastructure modification:

```text
false
```

Used when the final decision is normal or no safe action is required.

### `collect_host_diagnostics`

Type:

```text
local_diagnostics
```

Infrastructure modification:

```text
false
```

Collects local DevOps server diagnostics:

```text
hostname information
uptime
memory usage
disk usage
network interfaces
Prometheus service status
Grafana service status
```

### `collect_network_diagnostics`

Type:

```text
network_diagnostics
```

Infrastructure modification:

```text
false
```

Collects network-side diagnostic information without changing routers, switches or services.

Example checks:

```text
DevOps network interfaces
routing table
ping to selected OOB devices
Prometheus readiness
Prometheus target health query
```

### `refresh_monitoring_snapshot`

Type:

```text
local_command
```

Infrastructure modification:

```text
false
```

Re-exports the latest Prometheus snapshot using:

```text
monitoring/scripts/export-prometheus-snapshot.sh
```

This is useful after an anomaly or remediation attempt to refresh analyzer/dashboard evidence.

### `run_validation_gate`

Type:

```text
ansible_validation
```

Infrastructure modification:

```text
false
```

Runs the existing Ansible validation gate.

It can be used to confirm whether the topology is still valid after a detected anomaly.

### `restart_dmz_web`

Type:

```text
gns3_docker_restart
```

Infrastructure modification:

```text
true
```

Restarts only the DMZ Web/Nginx container on the GNS3 host.

This action requires:

```text
remediation_allowed = true
--mode apply
--confirm-apply
GNS3 host access
```

### `restart_dmz_dns`

Type:

```text
gns3_docker_restart
```

Infrastructure modification:

```text
true
```

Restarts only the DMZ DNS container on the GNS3 host.

This action requires:

```text
remediation_allowed = true
--mode apply
--confirm-apply
GNS3 host access
```

### `restart_dmz_services`

Type:

```text
gns3_docker_restart
```

Infrastructure modification:

```text
true
```

Restarts both DMZ Web and DNS containers on the GNS3 host.

This is used only when the final decision confirms a service-level anomaly and remediation is allowed.

## Auto Action Selection

When the runner is called with:

```text
--action auto
```

it chooses the action based on the final decision context.

General logic:

```text
normal decision
    → no_action

ML-only anomaly or suspicious signal
    → collect diagnostics only

SNMP device down
    → collect_network_diagnostics

unexpected interface down
    → run_validation_gate

interface errors/discards
    → collect_network_diagnostics

host CPU/memory/disk issue
    → collect_host_diagnostics

DMZ/Web/DNS validation or probe failure
    → restart_dmz_services, only if remediation is allowed
```

The runner does not blindly trust ML output. It uses the final merged decision and the rule-based analyzer’s detection reasons.

## Run in Plan Mode

Plan mode is the safest mode.

It shows what would happen without executing commands.

```bash
python cloud/analyzer/remediation/run_safe_remediation.py \
  --decision cloud/analyzer/outputs/final-decision.json \
  --mode plan
```

Expected healthy result:

```text
Selected action: no_action
Executed: False
Success: True
```

View generated files:


## Plan a GNS3 Service Restart

Planning a restart does not execute the restart.

```bash
python cloud/analyzer/remediation/run_safe_remediation.py \
  --decision cloud/analyzer/outputs/final-decision.json \
  --action restart_dmz_web \
  --mode plan \
  --gns3-host <GNS3_HOST_IP>
```

Example for DNS:

```bash
python cloud/analyzer/remediation/run_safe_remediation.py \
  --decision cloud/analyzer/outputs/final-decision.json \
  --action restart_dmz_dns \
  --mode plan \
  --gns3-host <GNS3_HOST_IP>
```

## Apply a GNS3 Service Restart

Only run this when the final decision allows remediation.

```bash
python cloud/analyzer/remediation/run_safe_remediation.py \
  --decision cloud/analyzer/outputs/final-decision.json \
  --action restart_dmz_web \
  --mode apply \
  --confirm-apply \
  --gns3-host <GNS3_HOST_IP>
```

If SSH requires a key:

```bash
python cloud/analyzer/remediation/run_safe_remediation.py \
  --decision cloud/analyzer/outputs/final-decision.json \
  --action restart_dmz_web \
  --mode apply \
  --confirm-apply \
  --gns3-host <GNS3_HOST_IP> \
  --gns3-user gns3 \
  --gns3-key ~/.ssh/<key_name>
```

The runner uses SSH to connect to the GNS3 host and restart only the selected Docker container.

## Full Local Remediation Test

From the repository root:

```bash
cd ~/pfe-repo
source .venv/bin/activate
```

Run the full analyzer flow first:

```bash
python cloud/analyzer/ml/collect_prometheus_window.py \
  --prometheus-url http://localhost:9090 \
  --duration-minutes 60 \
  --step 60s

python cloud/analyzer/ml/build_feature_dataset.py

python cloud/analyzer/ml/predict_anomaly.py \
  --input-csv cloud/analyzer/ml/data/features/latest_features.csv

./monitoring/scripts/export-prometheus-snapshot.sh

python cloud/analyzer/analyze_validation_artifacts.py \
  --input-dir /var/lib/pfe-dashboard/outputs \
  --metrics-dir monitoring/outputs/latest \
  --output-dir cloud/analyzer/outputs \
  --build-label local-remediation-test

  --rule-decision cloud/analyzer/outputs/decision.json \
```


## Safety Examples

### Case 1: Normal System

```text
final_status = normal
final_risk_score < 25
```

Action:

```text
no_action
```

### Case 2: ML-Only Anomaly

```text
rule_anomalous = false
ml_anomalous = true
```

Action:

```text
collect diagnostics only
```

Infrastructure restart is not allowed.

### Case 3: Rule-Based Anomaly

```text
rule_anomalous = true
ml_anomalous = false
```

Action:

```text
safe rule-based action or diagnostics
```

Remediation can be allowed if the final decision says so.

### Case 4: Confirmed Anomaly

```text
rule_anomalous = true
ml_anomalous = true
```

Action:

```text
strongest remediation candidate
```

Still requires:

```text
--mode apply
--confirm-apply
```

### Case 5: Infrastructure-Changing Action Without Permission

If the final decision contains:

```text
remediation_allowed = false
```

and the user tries:

```text
restart_dmz_web
restart_dmz_dns
restart_dmz_services
```

the runner blocks the action.

## Future Jenkins Integration

The remediation layer should be integrated into Jenkins as a controlled final stage.

Recommended Jenkins behavior:

```text
Always run remediation in plan mode by default.
Archive remediation-plan.json and remediation-report.txt.
Apply remediation only when explicit parameters are enabled.
```

Suggested Jenkins parameters:

```text
ENABLE_SAFE_REMEDIATION=false
REMEDIATION_MODE=plan
CONFIRM_APPLY=false
REMEDIATION_ACTION=auto
```

Safe default:

```text
REMEDIATION_MODE=plan
```

Apply mode should require:

```text
ENABLE_SAFE_REMEDIATION=true
CONFIRM_APPLY=true
```

The Jenkins pipeline should consume:

```text
cloud/analyzer/outputs/final-decision.json
```

and only allow infrastructure-changing remediation when:

```text
remediation_allowed = true
```
